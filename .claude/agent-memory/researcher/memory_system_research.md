# Claude Code Memory System Research

**Research Date**: 2026-03-19
**Scope**: Auto-memory, MEMORY.md indexing, lessons-learned patterns, context anchoring, self-healing protocol

## Memory System Architecture

### Two-Tier Memory Model

Claude Code implements a complementary two-tier memory system:

| Aspect | CLAUDE.md Files | Auto Memory (MEMORY.md) |
|--------|-----------------|------------------------|
| **Author** | You (human) | Claude (AI) |
| **Content** | Instructions & rules | Learnings & patterns |
| **Scope** | Project/User/Org | Per working tree |
| **Loaded** | Every session (full) | Every session (first 200 lines) |
| **Purpose** | Guide behavior | Capture discoveries |
| **Location** | `./CLAUDE.md`, `~/.claude/CLAUDE.md` | `~/.claude/projects/<project>/memory/MEMORY.md` |

### CLAUDE.md Scope Hierarchy

**Priority order (highest to lowest)**:
1. **Managed policy** — Organization-wide, enforced by IT/DevOps
   - macOS: `/Library/Application Support/ClaudeCode/CLAUDE.md`
   - Linux/WSL: `/etc/claude-code/CLAUDE.md`
   - Windows: `C:\Program Files\ClaudeCode\CLAUDE.md`
2. **Project instructions** — `./CLAUDE.md` or `./.claude/CLAUDE.md` (checked into git)
3. **User instructions** — `~/.claude/CLAUDE.md` (personal, all projects)

**Loading behavior**:
- Ancestor CLAUDE.md files (walking up directory tree) are loaded in full at session start
- Subdirectory CLAUDE.md files load on-demand when Claude reads files in those subdirectories
- Outside git repos, project root is used as anchor

### Auto Memory Storage

**Location pattern**: `~/.claude/projects/<project>/memory/`
- `<project>` = git repository path with `/` replaced by `-`
- Example: `/Users/john/projects/my-app` → `./-Users-john-projects-my-app/`
- All worktrees and subdirectories in same repo **share one auto memory directory**

**Directory structure**:
```
~/.claude/projects/<project>/memory/
├── MEMORY.md           # Concise index (loaded at session start)
├── debugging.md        # Topic file (loaded on demand)
├── api-conventions.md  # Topic file (loaded on demand)
└── ...
```

**Key constraint**: MEMORY.md has a **hard 200-line limit** on what loads at session start
- Content beyond line 200 is visible but not loaded automatically
- Topic files have no size limit (loaded on demand via file tools)
- CLAUDE.md files load in full regardless of size (though >200 lines consume more context)

**Custom storage**: Set `autoMemoryDirectory` in user/local settings to override default location:
```json
{
  "autoMemoryDirectory": "~/my-custom-memory-dir"
}
```

**Machine-local storage**: Auto memory is NOT shared across machines or cloud environments
- All worktrees in same git repo share the memory directory
- Worktrees on different machines have separate memories

### Memory File Organization Strategy

**MEMORY.md as index**:
- Contains concise references (under 200 lines)
- Links to topic files for detailed information
- Claude automatically moves detailed content into topic files when MEMORY.md approaches 200-line limit
- Acts as table of contents for the memory directory

**Topic file pattern**:
- Separate thematic files (debugging.md, patterns.md, architecture.md)
- Can be created manually or auto-generated by Claude
- Not loaded at session start (no token cost)
- Loaded on-demand when needed
- No size restrictions

## Auto Memory Lifecycle

### How Auto Memory Works

1. **Session Start**: First 200 lines of MEMORY.md injected into context
2. **Active Recall**: Claude reads/writes memory files during session
3. **Auto-Capture**: Claude decides what's worth remembering:
   - Build commands and configurations
   - Debugging insights and workarounds
   - Architecture decisions
   - Code style preferences
   - Workflow habits and patterns
   - Recurring issues and solutions
4. **Memory Persistence**: Changes written to filesystem during session
5. **Session End**: Context survives across sessions via MEMORY.md

### Enabling/Disabling Auto Memory

**Default**: Enabled as of v2.1.59 (Feb 26, 2026)

**Toggle options**:
1. Run `/memory` in session and use auto memory toggle
2. Set in settings: `"autoMemoryEnabled": false`
3. Environment variable: `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1`

### "Writing memory" / "Recalled memory" UI

- "Writing memory": Claude is updating MEMORY.md or topic files
- "Recalled memory": Claude is reading from memory directory
- Both operations are synchronous during session

### Audit and Edit

**Access memory**:
- Run `/memory` command to browse all CLAUDE.md and rules files
- Navigate to auto memory folder to view raw MEMORY.md and topic files
- Files are plain markdown — fully editable and deletable

## Lessons-Learned System

### Error Capture Automation

**Hook event**: `PostToolUseFailure`
- Fires when a tool (Bash, Read, WebFetch, etc.) fails
- Captures: tool name, input, error message, timestamp

**Automatic capture process** (via hook):
- Hook script (`lessons-learned-capture.sh`) appends new entry
- Entry format includes:
  - Tool name and error code/message
  - Input that caused error
  - Timestamp (UTC)
  - Status: `NEEDS_FIX` (auto-set)

**Example entry structure**:
```markdown
### Error: Bash failure (2026-03-15T14:22:00Z)
- **Tool:** Bash
- **Input:** `some-command`
- **Error:** Exit code 1 with message
- **Status:** NEEDS_FIX
```

### Fix Workflow

1. **Capture**: PostToolUseFailure hook captures error automatically
2. **Fix**: Claude fixes underlying issue in current session
3. **Document**: Claude updates entry in lessons-learned.md:
   ```markdown
   - **Status:** RESOLVED
   - **Fix:** Description of what resolved it
   - **Prevention:** How to avoid in future
   ```
4. **Recall**: Next session loads lessons-learned.md as rule, preventing repeat

### Pattern Detection & Rule Promotion

**Pattern detection triggers**:
- Same error type appears 3+ times in lessons-learned.md
- Related errors across multiple tool categories
- Environmental issues appearing in multiple contexts

**Promotion workflow**:
1. Identify pattern across multiple lessons entries
2. Create or update permanent rule file in `.claude/rules/`
3. Dedup/archive resolved lessons that have been promoted
4. Link from lessons-learned.md to new rule for traceability

**Example promotion**:
- Multiple `git add` failures on deleted files → update `.claude/rules/git-workflow.md`
- Repeated bash escaping issues → update `.claude/rules/code-style.md`

### Lessons-Learned File Management

**Maintenance strategy**:
- Keep under 500 lines (readability)
- Archive resolved patterns to separate file or delete
- Promote 3+ identical entries to permanent rules
- Review periodically (monthly) for dedup opportunities
- Use rotation: keep recent (30 days), archive older

**File structure**:
```markdown
# Lessons Learned - Auto-Captured

This file is automatically updated by hooks when errors occur.
Claude reads this at the start of each session to avoid repeating mistakes.

## Error Patterns and Fixes

### Error: [Category] ([Timestamp])
- **Tool:** Name
- **Input:** Command/input that failed
- **Error:** Message and code
- **Status:** NEEDS_FIX | RESOLVED
- **Fix:** [If RESOLVED] What fixed it
- **Prevention:** [If RESOLVED] How to avoid in future
```

## Context Anchoring System

### What is Context Anchoring?

Context anchoring refers to mechanisms that preserve critical information during `/compact` compaction events. Key anchored items:

1. **CLAUDE.md** — Fully survives compaction, re-injected fresh into context after compact
2. **Rules files** (`.claude/rules/*.md`) — Always active for matching file types
3. **MEMORY.md (first 200 lines)** — Auto-memory re-injected after compact
4. **Session metadata** — Project path, git state, environment

### Compaction Lifecycle

**Trigger**: Auto-compaction when context fills beyond ~200K tokens

**Process**:
1. **PreCompact hook fires** — Last chance to save state externally
2. **Compaction summarizes** conversation history (excluding anchored items)
3. **PostCompact hook fires** — React to compaction result
4. **CLAUDE.md re-injected** — Full file reloaded fresh
5. **MEMORY.md re-injected** — First 200 lines reloaded fresh
6. **Session continues** with reduced context + re-anchored instructions

### PreCompact and PostCompact Hooks

**PreCompact hook**:
- Fires **before** compaction happens
- Last opportunity to capture state
- Use cases:
  - Save current git diff to temporary file
  - Dump important variables to JSON
  - Snapshot build state or test results
  - Write a custom summary you'll read after compaction

**PostCompact hook**:
- Fires **after** compaction completes
- Cannot block or modify compaction
- Use cases:
  - Log the generated summary
  - Update external tracking (Slack notification, database)
  - Trigger downstream processes
  - Validate compaction quality

**Configuration** (in `.claude/settings.json`):
```json
{
  "hooks": {
    "PreCompact": [
      {
        "type": "command",
        "command": ".claude/hooks/save-state.sh",
        "timeout": 10
      }
    ],
    "PostCompact": [
      {
        "type": "command",
        "command": ".claude/hooks/log-compaction.sh"
      }
    ]
  }
}
```

### Context Anchoring Best Practices

1. **Use CLAUDE.md for critical rules** — Only persistent mechanism guaranteed to survive
2. **Keep CLAUDE.md concise** — <200 lines for best compliance
3. **Use `.claude/rules/` for detailed guidance** — No size limit, path-scoped
4. **Implement PreCompact for state capture** — Save important variables/diffs
5. **Use topic files in MEMORY.md** — Split by theme to stay under 200 lines
6. **Document anchoring expectations** — Make clear which info survives compaction

### Known Limitations

**What DOES survive compaction**:
- CLAUDE.md (full file, fresh injection)
- MEMORY.md (first 200 lines, fresh injection)
- Rules files (active for matching files)
- Session metadata (project, environment)

**What DOES NOT survive compaction**:
- Earlier conversation turns (summarized away)
- Context beyond 200 lines of MEMORY.md
- Temporary/inline instructions (given only in conversation)
- Variable state created during session (must be saved to MEMORY.md or file)

## Self-Healing Protocol

### Error → Fix → Prevent Loop

**Cycle**:
1. **Error occurs** → Tool fails (Bash, Read, WebFetch, etc.)
2. **Capture** → PostToolUseFailure hook appends to lessons-learned.md
3. **Fix** → Claude fixes underlying issue in current session
4. **Document** → Claude updates entry: Status: RESOLVED, Fix:, Prevention:
5. **Recall** → Next session loads lessons-learned.md as rule file
6. **Prevent** → Claude avoids repeating the mistake

### Integration Points

**Hook system**:
- **Event**: `PostToolUseFailure`
- **Handler**: Bash script (lessons-learned-capture.sh)
- **Input**: JSON with tool name, error, input
- **Output**: Append to `.claude/rules/lessons-learned.md`

**Rule system**:
- **File**: `.claude/rules/lessons-learned.md`
- **Scope**: Global (no path restriction)
- **Loading**: Automatic at session start
- **Format**: Markdown list of error patterns and fixes

**Memory system**:
- **Integration**: Lessons can reference MEMORY.md for deeper context
- **Cross-reference**: "See debugging.md for related issues"
- **Topic linking**: Promote lessons to dedicated topic files

### Pattern Promotion Strategy

**Trigger**: 3+ entries with same root cause or related errors

**Promotion process**:
1. Create new `.claude/rules/<topic>.md` file
2. Document pattern, causes, and preventive measures
3. Add reference from lessons-learned.md to new rule
4. Archive/remove promoted entries from lessons-learned.md
5. Update entry with: `**Promoted to**: .claude/rules/<topic>.md`

**Example**: Git workflow errors
```markdown
# lessons-learned.md entry
- **Promoted to**: .claude/rules/git-workflow.md
  Repeated `git add` failures on deleted files → now documented in permanent rule
```

### Best Practices

1. **Capture comprehensively** — Every tool failure should be logged
2. **Fix immediately** — Don't defer debugging; resolve in current session
3. **Document the why** — Prevention only works if you understand root cause
4. **Rotate lessons** — Keep under 500 lines; archive/promote regularly
5. **Cross-link rules** — Reference related rules in prevention statements
6. **Test the fix** — Verify prevention works in same session

## Memory Types & Scoping

### Memory Type Framework

From Claude Code documentation, memory files support four types:

1. **User memories** (`user` type)
   - Location: `~/.claude/agent-memory/<agent>/user_*.md`
   - Scope: About user role, goals, responsibilities, knowledge
   - When to save: User profile details relevant to collaboration
   - Example: "User is data scientist focused on observability/logging"

2. **Feedback memories** (`feedback` type)
   - Location: `~/.claude/agent-memory/<agent>/feedback_*.md`
   - Scope: Guidance on how Claude should approach work
   - When to save: User corrections ("don't do X") OR confirmations ("yes, keep doing Y")
   - Structure: Rule + Why + How to apply
   - Example: "Never mock database in tests — we got burned when mocks passed but prod failed"

3. **Project memories** (`project` type)
   - Location: `~/.claude/agent-memory/<agent>/project_*.md`
   - Scope: Ongoing work, goals, deadlines, stakeholders
   - When to save: Who is doing what, why, by when
   - Note: Convert relative dates to absolute (e.g., "Thursday" → "2026-03-19")
   - Structure: Fact/decision + Why + How to apply
   - Example: "Merge freeze on 2026-03-19 for mobile release"

4. **Reference memories** (`reference` type)
   - Location: `~/.claude/agent-memory/<agent>/reference_*.md`
   - Scope: External system pointers
   - When to save: Where to find information (Linear project, Slack channel, Grafana board)
   - Example: "Pipeline bugs tracked in Linear project 'INGEST'"

### Agent Memory Directory

**Location**: `~/.claude/agent-memory/<agent-name>/`
- Separate memory per subagent
- Can be committed to git for team sharing
- Persistent across conversations with that agent

**Format**: MEMORY.md index pattern (up to 200 lines loaded at start)
```
~/.claude/agent-memory/researcher/
├── MEMORY.md              # Index
├── user_profile.md        # User type
├── feedback_testing.md    # Feedback type
├── project_status.md      # Project type
└── reference_external.md  # Reference type
```

**Frontmatter format** (for memory files):
```markdown
---
name: Memory name
description: One-line description for relevance
type: user | feedback | project | reference
---

Content here...
```

### MCP-Backed Memory

**Protocol**: Model Context Protocol (MCP)
- **Package**: `@modelcontextprotocol/server-memory`
- **Storage**: SQLite database (local)
- **Transport**: Stdio
- **Access**: Claude Code and Claude Desktop via same database file

**Use case**: Semantic memory for long-term knowledge bases
- Shared knowledge graphs across sessions
- Tool-based memory access (read/write via MCP)
- Structured data (not just markdown)

## Key Findings & Recommendations

### Critical Insights

1. **200-line hard limit is load-bearing** — Only MEMORY.md first 200 lines load automatically; rest requires manual read
2. **Context survives compaction only via anchoring** — CLAUDE.md and rules files are the only guaranteed persistent mechanisms
3. **Pattern detection requires manual promotion** — No automated mechanism yet; requires human judgment to detect 3+ similar errors and create permanent rules
4. **Compliance is probabilistic, not enforced** — CLAUDE.md is context not enforcement; expect 70-80% compliance; use hooks for hard constraints
5. **Auto memory decides what to capture** — Claude selectively saves discoveries; doesn't save every detail

### Compliance Patterns (2026 Research)

- **Focused 30-line rules > comprehensive 200-line files** — Each instruction gets more attention weight in focused files
- **Positive instructions > negative rules** — "Use pnpm" works better than "Don't use npm"
- **Rule + Hook hybrid** — Combine instructions (suggestions) with hooks (enforcement) for optimal results
- **Specificity drives compliance** — "Use 2-space indentation" >> "Format code properly"

### Context Management Dos and Don'ts

**Do**:
- Keep CLAUDE.md under 200 lines
- Use `.claude/rules/` for detailed per-file-type guidance
- Put critical rules in CLAUDE.md (survives compaction)
- Use topic files in MEMORY.md for detailed learnings
- Implement PreCompact hooks to save state
- Cross-reference rules in MEMORY.md

**Don't**:
- Assume instructions beyond 200 lines in MEMORY.md load automatically
- Rely solely on inline conversation instructions for persistence
- Create CLAUDE.md files larger than 400 lines (sharp compliance drop-off)
- Store critical state only in conversation (will be lost after compaction)
- Forget to run lessons-learned file maintenance (rotate, archive, promote)
